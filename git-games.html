In Reflection

Throughout the course I wrote prompted reflections on what were were learning and the merits of various aspects of the version control software. Here are some important excerpts I feel exemplify my learning:

How using version control allows us to be more confident and innovative
By using Git, we obtain freedom to try big changes without worrying about how to revert back to previous versions. This way we don't have to worry about breaking our code. Also, if we do break our code and don't (yet, perhaps) know how to fix it, we can revert to a previous state and make a new branch, leaving the broken code wait for us until we are ready to deal with it. This interview from Lewis Kaneshiro helps explain how Git has helped him become a prolific programmer. I'm honestly excited to use Git in my personal projects and am excited by the idea of job in a collaborative enviroment that utilizes version control.

While this is exciting and Git has great power, learning to use Git is not a trivial experience. Below I highlight some of the tactics for using Git commands to identify when a bug was introduced into a program and how to identify the lines of code that fix it.

How git commands "git log" and "git diff" to view the history of files
By using the command git log you view the committed changes that have been logged. Each log contains a comment describing what changed in that commit. If you think a certain commit may have introduced a bug into your program, you can "step into" certain log states by using the git checkout ID command where ID is the commit ID from the commit log. If you are interested in what specific code changes during a commit, you coukd compare it to the previous commit. To do this you use the git diff command along with the IDs provided by the log. Note that in your command prompt, copy and paste hotkeys may not be the same as what you are used to! (apparently it's crtl/shift + ins in my Git Bash MINGW64 command prompt).

Expanding on the functionality of Git, we learned the nuanced differences between our working directory, Git repository, and the staging area. At first one might think, "Is it really necessary to have these different spaces? Why not just automatically update my repository when I make a change in the working directory?" Simply put, the more control we have the more flexibility we have in our work flow. What would happen if Git committed a change in the middle of a coding line? It's best for us to commit our own changes at meaningful points.

Repository, working directory, and staging area, oh my!
The working directory contains the physical files on your hard drive. The repository contains the data files for the history of commits/branches. The staging area lets you take items that you have changed in your working directory and then commit changes. The staging area lets you commit any number of files (one at a time or multiple files together) depending on which files you are ready to commit, allowing for flexibility.

Note on Commands:
When there no changes to stage, the staging area contains the most recent commit. This helps us utilize these commands:
  • git diff compares the working directory to the staging area
  • git diff --staged compares the staging area to the most recent commit
  • git diff ID1 ID2 compares the two commits with ID1 and ID2
For etiquette on commits comments, check out the Udacity Git Commit Message Style Guide
